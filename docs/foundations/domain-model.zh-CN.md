# Gaia 领域模型

## 目的

本文档定义 Gaia 基础文档使用的共享核心词汇表。

它的职责是在后续文档定义以下内容之前锁定最小领域术语集：

- 包模式
- 包文件格式
- 全局图整合
- 概率语义

## 边界

本文档仅定义词汇和建模边界。

它不定义：

- 包文件布局
- 规范化或审查算法
- 服务器 API 契约
- 图传播、先验、信念或 BP

## 核心层

Gaia V1 使用三个概念层：

1. 全局 `closure` —— 自包含、可重用的知识对象
2. 本地 `module` —— 将闭包组织成带导入和导出的链的连贯单元
3. 本地 `package` —— 导出选定闭包的模块集合

设计遵循受函数式编程启发的**状态-行动模型**：

- 闭包是**状态** —— 可以独立传递的自包含对象
- 推理是**行动** —— 在模块链中连接闭包的本地推理步骤

## 共享知识对象

### `closure`

`closure` 是全局可识别、自包含的知识对象。

名称来自函数式编程：就像捕获自由变量并可以独立传递的 FP 闭包，知识闭包携带其内容和元数据，可以在不知道创建它的链的情况下被导出、导入和引用。

V1 刻意保持闭包类型集较小：

- `claim`
- `question`
- `setting`
- `action`

这是本地包工具和后续服务器端摄取都使用的公共基础。

### `claim`

`claim` 是可真伪的陈述或可重用的结果对象。

示例：

- 科学陈述
- 声明式差距陈述
- 可重用的代码结果
- 可重用的 Lean 定理/证明结果

`claim` 是类似陈述内容的默认类型。

### `question`

`question` 是询问对象。

它不是可真伪的陈述。

示例：

- "为什么现象 X 会发生？"
- "实现 Y 能被证明正确吗？"

### `setting`

`setting` 是设定上下文的对象，决定后续推理应如何被解释或执行。

V1 使用 `setting` 统一：

- 定义
- 逻辑设定
- 执行环境
- 实验环境

### `action`

`action` 是自包含、可重用的过程描述。

它代表一个过程，如推理方法、工具或其他规范化过程。行动闭包描述过程**是什么**；模块链中的 `inference` 条目描述**如何**在特定上下文中应用它。

示例：

- 推理方法描述
- 工具调用规范
- 其他显式过程描述

## 本地推理结构

### `inference`

`inference` 是模块链中的本地推理步骤。它通过填补逻辑空白、提供解释或描述行动如何被应用来连接闭包。

与闭包不同，推理**不是**自包含的 —— 它们依赖于链中的周围上下文。它们从不被导出或从模块外部引用。

在 FP 类比中：闭包是**值**，推理是**lambda**。推理可以是：

- **匿名的** —— 描述推理的纯文本（像 lambda）
- **命名的** —— 引用可重用的 `action` 闭包（像命名函数应用）

这将 `action` 闭包（函数定义）连接到 `inference` 条目（函数调用）。行动描述过程**是什么**；命名推理描述该过程**如何**在特定上下文中被应用。

### 链模型

模块的 `chain` 定义叙事阅读顺序：应用于值的 lambda 的顺序组合。

```text
closure → inference → closure → inference → closure
  v₀    →    λ₁    →   v₁    →    λ₂    →   v₂
```

当两个闭包之间的逻辑转换是平凡的或局部明显的时，推理可以省略 —— 链中两个相邻闭包意味着平凡转换。

### 跨模块依赖

模块通过 `imports` 声明跨模块依赖，依赖角色今天编写为 `dependency: direct/indirect`：

- **direct** —— 语义角色 `premise`。如果导入的闭包是错的，该模块的结论也可能是错的。跨包边界，只有导出的知识可以这样使用。
- **indirect** —— 语义角色 `context`。导入的闭包是相关的背景，但该模块的结论可以独立成立。非导出的外部闭包仍可以显式引用，但只能在此上下文角色中。

这些是本地模块关系，而非全局闭包属性。同一个闭包可以在一个模块中是直接依赖，在另一个模块中是间接依赖。

## 模块和包组织

### `module`

`module` 通过闭包和推理链将闭包组织成连贯单元。它从其他模块导入闭包并导出选定的闭包作为其公共接口。这类似于 Rust 或 Julia 中的模块：它组织相关逻辑，声明其依赖（`imports`），并暴露公共接口（`exports`）。

模块有一个可选的 `role`，描述其在包中的目的：

- `reasoning` —— 通过前提、推理和结果建立结论
- `setting` —— 建立共享上下文（定义、环境、假设）
- `motivation` —— 建立为什么进行研究
- `follow_up_question` —— 建立未来工作的开放问题
- `other`

模块角色取代了包上单独编辑字段的需要。研究工作的动机表示为 `motivation` 模块；共享定义成为 `setting` 模块；开放问题成为 `follow_up_question` 模块。结构本身承载编辑意图。

### `package`

`package` 是模块的可重用容器。它从其模块导出选定的闭包作为包的公共接口。

典型示例包括：

- 论文
- 研究包
- 结构化笔记
- 项目单元

包的 `modules[]` 列表顺序定义了包的推荐阅读顺序（叙事排序）。包的 `exports[]` 列表是从其模块中选定的 closure_id 子集 —— 该包向外界提供的关键结果。

## 重要的不等价性

以下内容不应视为等价：

- `claim` 和 `question`
- `claim` 和 `setting`
- `action`（闭包类型）和 `inference`（本地链条目）
- 全局 `closure` 身份和本地链出现
- 叙事排序和逻辑依赖

示例：

- 陈述形式的差距是 `claim`，而非 `question`
- 定义是 `setting`，而非通用 `claim`
- `action` 闭包描述过程**是什么**；`inference` 描述它**如何**在本地被应用
- 模块链中相邻闭包不意味着逻辑依赖 —— 所有依赖通过 `imports` 声明

## 推迟的区别

V1 刻意不将闭包系统拆分为更详细的认知类型，如：

- `observation`
- `assumption`
- `conjecture`

这些区别在后续图整合、概率语义或审查策略中可能变得重要，但对于最小共享包基础不需要。

目前：

- 类观察内容建模为 `claim` 加来源和支持资源
- 类假设内容建模为 `claim` 或 `setting`，取决于它是陈述式的还是设定式的

## 当前规则

当新设计需要新名词时，优先：

1. 如果已有 `claim`、`question`、`setting` 或 `action` 适合，则重用
2. 将本地依赖语义放在模块 `imports` 上，而非全局闭包类型
3. 推迟更细的认知区别，直到图或概率语义需要
