# Gaia 产品范围

## 目的

本文档定义了 Gaia 的产品定位、理论基础和 `main` 分支上的当前基线。

它的职责是：

- 陈述理论基础 —— 为什么存在 Gaia 以及为什么现有工具无法替代它
- 陈述已决定的产品方向
- 陈述 `main` 上当前发布的内容
- 陈述尚未发布但在路线图上的内容

## 理论基础

### 目标：大规模知识模型

Gaia 的目标是构建一个**大规模知识模型（LKM）** —— 一个十亿级规模的推理超图，其中命题是节点，推理关系是超边。LKM 由云端的 AI 代理自动构建，而非人工手动创建。

这需要一种**机器可读、机器可写的知识表示**，支持：

1. **结构化推理** —— 不是自由文本，而是通过显式推理步骤连接的类型化知识对象（主张、问题、设定、行动）
2. **概率信念** —— 知识不是真/假，而是带有信念度，因为科学知识本质上是确定性的
3. **可组合模块** —— 知识必须可以打包、导出、导入和组合，就像编程语言中的代码
4. **自动推理** —— 当新证据到达时，系统必须自动在图中传播信念

这些要求共同定义了一种编程语言。Gaia 就是这种语言。

### Gaia 作为概率函数式编程语言

Gaia 是一种**概率函数式编程语言**，专门用于知识表示和认知推理。

它遵循概率编程语言的标准架构：一个**确定性宿主语言**加上一个**概率层**。

| 层 | 它提供什么 | PL 类比 |
|---|-----------|---------|
| **V1 — 确定性 FP 核心** | 知识（值）、推理（lambda）、链（组合）、模块（带导入/导出）、包 | Haskell, OCaml |
| **V3 — 概率层** | 先验、依赖角色（`direct`/`indirect` 表面语法；`premise`/`context` 语义）、信念传播（推理） | Church 的 `flip`/`observe`、Hakaru 的 `measure` monad、Pyro 的 `sample`/`observe` |

理论定位：

- **Pólya**（《数学与合情推理》，1954）—— 推理延伸超出演绎证明到合情推理
- **Jaynes**（《概率论：科学的逻辑》，2003）—— 概率是合情推理的逻辑，是演绎逻辑到信念度的推广
- **Cox 定理** —— 任何一致的合情推理系统都与概率论同构

Gaia 渴望成为**合情推理的 Curry-Howard**：正如函数式编程语言（Haskell、Lean）基于证明与程序之间的 Curry-Howard 对应，Gaia 将其从演绎确定性扩展到合情信念。这是一个开放的研究方向，而非既定定理 —— 概率计算的完整 Curry-Howard 对应仍是活跃的研究领域（参见 "Curry and Howard Meet Borel", LICS 2022）。

### 为什么现有工具无法替代 Gaia

**为什么不是现有的概率编程语言（Pyro、Stan、Church、Hakaru）？**

这些语言建模**统计概率** —— 随机变量的分布用于数据建模。Gaia 建模**认知概率** —— 命题真理的信念度。

| | 统计概率（Pyro/Stan） | 认知概率（Gaia） |
|---|---|---|
| 概率关于什么 | 随机变量（数值） | 命题（知识对象） |
| 概率意味着 | 结果的频率/测度 | 真理的信念度 |
| 条件化于 | 观测数据 | 依赖角色（`direct` / `indirect`；语义 `premise` / `context`） |
| 图模型 | DAG（贝叶斯网络） | 超图（多前提 → 多结论） |
| 推理计算 | 后验分布 | 命题上的信念分数 |
| 推理算法 | MCMC、变分推理 | 超图上的循环 BP |

数学基础相同（贝叶斯概率），但领域结构根本不同 —— 就像 SQL 和 Haskell 都基于集合论，但 SQL 存在是因为关系数据需要自己的语言。

**为什么不是现有的函数式语言（Haskell、OCaml）？**

它们提供演绎逻辑层但没有内置概率推理。你可以在 Haskell 之上构建 Gaia，但你需要添加整个概率层（先验、信念传播、条件语义、超图推理）—— 此时你已经构建了一种新语言。

**为什么不是现有的知识图谱（Neo4j、OWL、RDF）？**

它们提供图存储和确定性查询，但没有概率推理。它们可以告诉你"A 连接到 B"，但不能告诉你"给定 A 的证据，你应该在多大程度上相信 B？" Gaia 使用图数据库（Neo4j、Kuzu）作为存储后端，但知识表示和推理层才是让 Gaia 成为语言而不仅仅是数据库的原因。

### Gaia 的独特组合

Gaia 结合了三种现有工具无法同时提供的能力：

1. **函数式知识结构**（V1）—— 知识对象、推理、链、模块、包 —— 一种受 Haskell/OCaml 模块系统启发的类型化、可组合知识表示
2. **认知概率推理**（V3）—— 先验、信念、依赖角色、矛盾/撤回语义 —— 基于 Jaynes 的概率即逻辑传统
3. **超图信念传播** —— 从知识包结构导出的因子图上的循环 BP，计算整个 LKM 中自洽的信念

这种组合实现了核心产品：**AI 代理编写知识包（概率程序），云端运行信念传播（后验推理），结果是一个大规模知识模型，其中每个命题都带有校准的信念度。**

## 产品方向（已决定）

Gaia 是**CLI 优先，服务器增强**。

- **CLI 是主要产品** —— AI 代理和研究人员通过 CLI 与 Gaia 交互，在本地工作，零服务器依赖
- **目标本地流水线**是 `build -> self-review skill -> graph-construction skill -> infer -> publish`
- **服务器提供四个可选增强服务：**
  1. 知识整合 —— 将批准的包合并到全局大规模知识模型
  2. 全局搜索 —— 跨包向量 + BM25 + 拓扑搜索
  3. 同行审查和注册表整合 —— 独立审查、编辑决策、身份分配
  4. 大规模 BP —— GPU 集群上的十亿节点信念传播

目标共享交互路径是：**CLI → git push / publish → 同行审查 → 反驳/编辑循环 → 合并或拒绝**（类似于学术出版工作流）。

用户可以完全离线使用 CLI 工作。服务器是可选的注册表和计算后端，而非先决条件。

## `main` 上的当前基线

`main` 上当前发布的内容：

- 后端推理图服务（FastAPI）—— 这是服务器端
- 用于浏览、图探索和提交工作流的仪表盘前端
- 带有 Neo4j 和 Kuzu 实现的 GraphStore ABC
- 类型感知的信念传播（矛盾、撤回边）
- **带 8 个命令的 CLI**（`init`、`build`、`review`、`infer`、`publish`、`show`、`search`、`clean`）—— 在 PR #63 中发布
- **目标架构文档现在只将 `build`、`infer` 和 `publish` 视为核心 CLI 流水线命令**。发布的 `gaia review` 命令仍是本地自审查边车的兼容路径。
- **Gaia Language** —— 每模块 YAML，包含知识对象、链和 `dependency: direct/indirect` 引用
- **推理引擎移至 `libs/inference/`** —— 与服务器解耦的本地信念传播
- **构建输出** —— 用于 LLM 审查的每模块 Markdown

尚未发布但在路线图上的内容：

- `gaia publish --server`（直接服务器发布，无需 git）
- GitHub webhook 集成用于发布时同行审查
- 反驳/编辑发布循环
- 注册表端 `CanonicalBinding` + `GlobalInferenceState` 端到端流程
- 跨包依赖解析和 `gaia.lock`
- 共享知识包契约（在此基础工作中标准化）

## 当前支持的产品界面

### 1. HTTP 服务器

当前发布的服务器界面是从 `services/gateway/` 暴露的 FastAPI 服务。

当前 API 区域：

- 提交提交、审查、合并和检索
- 节点、超边、矛盾、子图和统计的读取 API
- 节点和超边的搜索 API
- 提交、读取、子图和搜索流程的批量 API
- 异步状态和结果检索的作业 API

这是今天主要的外部可寻址界面。在 CLI 优先定位下，服务器成为 CLI 发布到的注册表和计算后端。

### 2. 仪表盘前端

当前前端产品是 `frontend/` 中的 React 仪表盘。

当前 UI 界面包括：

- 仪表盘着陆页
- 数据浏览器
- 图探索器
- 节点和边详情页
- 提交面板

前端应被视为当前服务器 API 的客户端，而非具有独立领域契约的独立产品线。

### 3. 服务器端图和存储运行时

Gaia 目前发布了一个服务器端存储/运行时栈，包括：

- 用于节点内容和元数据的 LanceDB
- 带有本地 LanceDB 支持实现的向量搜索层
- 图后端抽象（`GraphStore`）
- Neo4j 和 Kuzu 图后端实现

`main` 上的当前后端模式：

- `graph_backend="neo4j"`：默认面向服务器的图后端
- `graph_backend="kuzu"`：当前代码中可用的嵌入式图后端
- `graph_backend="none"`：图操作不可用的情况的降级模式

重要边界：

代码中存在多个图后端并不意味着 Gaia 有完全规范的产品契约保证后端对等。能力矩阵仍需明确记录。

### 4. 本地开发者工作流

Gaia 目前支持基于以下内容的本地开发工作流：

- 可编辑的 Python 安装
- 种子本地数据库
- 运行 FastAPI 服务器
- 运行 Vite 前端
- 运行测试套件

这是开发和验证工作流，不是正式的终端用户本地产品体验。

## 明确不在当前产品范围内

以下内容不应被描述为 `main` 上当前的 Gaia 产品能力，除非它们已单独合并和记录。

### 1. CLI/包管理器产品 —— 现已发布（PR #63）

CLI 已在 `main` 上发布，带 8 个命令：

| 命令 | 用途 |
|-----|------|
| `gaia init` | 初始化知识包 |
| `gaia build` | 解析 YAML，解析引用，降低包本地图 IR → `.gaia/build/` + `.gaia/graph/` 工件 |
| `gaia review` | 为本地自审查边车发布的兼容性辅助 |
| `gaia infer` | 从本地图 IR + 本地审查边车派生本地参数化，然后运行本地信念传播 |
| `gaia publish` | 发布到 git 或本地数据库（LanceDB + Kuzu） |
| `gaia show` | 显示知识对象详情 + 连接的链 |
| `gaia search` | 在本地 LanceDB 中搜索已发布节点 |
| `gaia clean` | 移除构建工件（`.gaia/` 目录） |

注意：目标流水线语义不再将 `gaia review` 视为最小核心命令之一。在目标架构中，自审查是代理技能；发布的命令是当前 `main` 上的桥梁。原始 RFC 还包括 `gaia claim` —— 这已被声明式 YAML 创作（带有知识对象和链的每模块 YAML 文件）取代。`gaia.lock` 和跨包依赖解析仍被推迟。

仍未发布：

- `gaia publish --server`（直接服务器发布）
- GitHub webhook 集成
- 发布时同行审查/反驳/编辑循环
- `gaia.lock` / 跨包依赖解析

### 2. 生产 ByteHouse 支持的部署

配置仍包含面向 ByteHouse 的字段，但 ByteHouse 不是完全实现的当前产品存储路径。

在改变之前，ByteHouse 应被视为计划或保留的，而非支持的当前部署。

### 3. 完全规范的后端可互换性

Gaia 现在有多个图后端实现，但还没有完全记录、稳定的产品级保证每个后端完全相同地支持每个图相关特性。

在能力矩阵存在之前，应避免关于后端对等的声明。

### 4. GitHub 原生审查/发布生态系统

Gaia 目前未发布完整的产品故事用于：

- GitHub 机器人审查
- PR 原生推理包验证
- 联邦或社区审查工作流
- 发布到 Git 作为支持的终端用户路径

这些仍是设计方向，而非当前基线能力。

## 产品定位总结

Gaia 是一个**CLI 优先、服务器增强**的大规模知识模型平台。

- **CLI** —— 创建、构建、审查和发布知识包的主要产品界面
- **服务器** —— 提供知识整合、全局搜索、同行审查/身份分配和大规模 BP 的可选注册表
- **仪表盘** —— 用于探索服务器端知识图谱的浏览器 UI

当前 `main` 发布了服务器、仪表盘和 CLI。

## 对未来工作的影响

大型新工作应以以下方式之一构建：

1. 构建 CLI 产品界面（主要产品）
2. 将服务器扩展为 CLI 的注册表和计算后端
3. 加强 CLI 和服务器都依赖的共享基础

这意味着：

- 共享契约（知识包模式、领域词汇表）是最高优先级的基础工作
- CLI 架构应驱动设计决策，而非事后考虑
- 服务器工作应专注于四个增强服务，而非成为唯一的产品界面

## PR 和文档规则

编写文档或审查 PR 时：

1. 如果特性是当前产品基线的一部分，可以将其描述为当前行为。
2. 如果已在代码中实现但尚未完全规范为产品契约，请明确指出限制。
3. 如果只是计划中的，请将其标记为提案、路线图或未来工作。
4. 不要让设计文档默默地重新定义当前产品基线。

## 已决定的问题

这些已解决，不应重新讨论：

1. **CLI 优先还是服务器优先？** → CLI 优先，服务器增强。
2. **主要交互路径？** → CLI 本地流水线（`build -> self-review -> graph-construction -> infer -> publish`）→ 共享同行审查/编辑循环 → 合并或拒绝。
3. **Kuzu 角色？** → CLI 的嵌入式图后端（本地、零配置）。Neo4j 是服务器端后端。
