## 0. Executive Summary

Gaia IR v6 keeps the current core shape **unchanged**:

```
Knowledge    — unchanged schema
Operator     — unchanged schema
Strategy     — unchanged schema
```

and adds:

```
Knowledge    — new type "context", grounding metadata, parameterized claim rendering
ReviewManifest / Review  — package-level review layer (new)
```

The main semantic change is not structural but interpretive:

```
Old interpretation:
  Strategy prior / warrant prior on helper claims

New interpretation:
  Strategy has no probability. Uncertainty is on explicit premise Claims.
  ReviewManifest makes qualitative accept/reject/needs-inputs decisions.
  Numeric warrant strength is produced by the accepted lowering policy, not by Review.
```

### Core rules

1. **Only `Claim` carries epistemic probability.** Setting, Context, Question do not.
2. **Strategy carries no probability.** Uncertainty is expressed through explicit premise Claims.
3. **Operator remains deterministic** and continues to produce `conclusion` helper Claims.
4. **Generated helper Claims are qualitative audit targets** unless their owning Strategy/Operator is reviewed and accepted.
5. **`Review` is a review record in `ReviewManifest`**, not a Claim and not a probability variable.
6. **Statistical evidence** uses existing `Strategy(type="infer")` with CPT encoding P(E|H) and P(E|¬H).

## 1. Knowledge Extensions

### 1.1 New Knowledge type: `context`

```python
Knowledge.type ∈ {
    "claim",
    "setting",
    "question",
    "context",      # NEW
}
```

`context` stores raw, not-yet-formalized text or artifact excerpts (paper paragraphs, lab notes, dashboard data). Does not participate in BP. Provides traceability for later formalization.

### 1.2 Parameter.value

Existing `Parameter` gains a bound value field:

```python
class Parameter:
    name: str
    type: str
    value: JsonValue | None = None   # NEW
```

The `value` participates in the content hash of a ground parameterized Claim.

When a parameter's type is a Knowledge type (e.g., `Setting`, `Claim`), the `value` stores the referenced node's QID:

```python
Parameter(name="experiment", type="Setting", value="github:my_package::exp_123")
```

### 1.3 Grounding metadata

Grounding is stored in `metadata.grounding`, not as a top-level field:

```python
metadata = {
    "grounding": {
        "kind": "assumption" | "source_fact" | "definition" | "imported" | "judgment" | "open",
        "rationale": "...",
        "source_refs": ["..."],
    }
}
```

A root Claim with a non-default prior should have grounding metadata or incoming Strategy support. Grounding does not affect BP directly.

### 1.4 Parameterized Claim rendering

The original docstring template is stored in `metadata.content_template`:

```python
metadata = {
    "content_template": "[@experiment] recorded {ctrl_k}/{ctrl_n} control conversions.",
}
```

`Knowledge.content` stores the rendered text after parameter substitution. Content hash uses the template + bound parameter values for stable identity.

## 2. Strategy — No Schema Change

The existing Strategy schema is sufficient for v6:

```python
class Strategy(BaseModel):
    strategy_id: str | None = None
    scope: str
    type: StrategyType
    premises: list[str]
    conclusion: str | None = None
    background: list[str] | None = None
    steps: list[Step] | None = None
    metadata: dict[str, Any] | None = None
```

### 2.1 How v6 Lang constructs map to existing fields

| Lang concept | IR field |
|---|---|
| `rationale=` | `steps=[Step(reasoning=rationale)]` |
| observation vs derivation distinction | `metadata.pattern = "observation"` or `"derivation"` |
| compute function info | `metadata.compute = {function_ref, code_hash, ...}` |
| P(E\|H), P(E\|¬H) for infer | `StrategyParamRecord.conditional_probabilities = [p_e_given_not_h, p_e_given_h]`. Always 2 entries. `premises=[H]`, `conclusion=E`. Assumptions go in `background` (not in BP); reviewed via ReviewManifest. |
| helper claims (warrants) | Generated by formalize as Operator conclusions, existing mechanism |

### 2.2 StrategyType — No Change

Existing types are reused:

| v6 Lang verb | StrategyType | Notes |
|---|---|---|
| `derive()` | `deduction` | Existing FormalStrategy lowering |
| `observe()` with premises | `deduction` | `metadata.pattern = "observation"` |
| `observe()` no premises | `deduction` | Reviewable root-observation Strategy with `premises=[]`; adds Grounding to the Claim; lowering does not create a support edge from non-existent premises |
| `compute()` | `deduction` | `metadata.pattern = "computation"`, `metadata.compute = {...}` |
| `infer()` | `infer` | Existing; `premises=[H]`, `conclusion=E`, CPT = `[p_e_given_not_h, p_e_given_h]` |

### 2.3 Operator — No Change

Existing Operator schema and types (conjunction, disjunction, equivalence, contradiction, complement, implication) are unchanged.

`equal()` compiles to `Operator(type="equivalence")`.
`contradict()` compiles to `Operator(type="contradiction")`.

## 3. ReviewManifest (New)

### 3.1 Separation from semantic graph

```python
class GaiaPackageArtifact:
    graph: LocalCanonicalGraph
    review: ReviewManifest | None = None   # NEW
```

Review status is deliberately separated from the canonical semantic graph. Review status can change without changing the semantic content.

### 3.2 Review schema

```python
class ReviewStatus(StrEnum):
    UNREVIEWED = "unreviewed"
    ACCEPTED = "accepted"
    REJECTED = "rejected"
    NEEDS_INPUTS = "needs_inputs"

class Review:
    review_id: str
    action_label: str           # references Lang Action label (human-readable)
    target_kind: Literal["strategy", "operator"]
    target_id: str              # Strategy.strategy_id or Operator.operator_id
    status: ReviewStatus
    audit_question: str         # auto-generated from target type / pattern
    reviewer_notes: str | None = None
    timestamp: str
    round: int                  # supports multi-round review
```

### 3.3 Multi-round review

A single Strategy or Operator can have multiple Review records across review rounds. The latest Review's status determines whether the target participates in inference.

### 3.4 Default behavior

**Unreviewed Strategy/Operator targets do NOT participate in inference** (pessimistic default). A target must be explicitly accepted before it contributes to belief propagation.

### 3.5 Review has no probability

```
Review is procedural review state, not epistemic uncertainty.
```

Review records therefore do not carry priors, likelihoods, or calibration values. They only decide whether a Strategy/Operator is accepted for lowering. If a reviewer finds an issue, the fix is qualitative and structural: add missing premise Claims, adjust Claim priors in the parameterization layer, or mark the Review as `needs_inputs`.

Accepted review status authorizes the existing lowering policy to assign runtime semantics:

- accepted `derive`/`observe`/`compute` FormalStrategy instances participate in lowering
- accepted `equal`/`contradict` Operators participate in lowering
- deterministic/formal warrant helpers use the backend's standard assertion/default policy
- accepted `infer` uses its explicit CPT record
- unreviewed/rejected/needs-inputs Strategy/Operator targets are excluded from inference

### 3.6 Auto-generated audit questions

Audit questions are automatically generated from target type, using `[@...]` reference templates:

| Target type / pattern | Audit question template |
|---|---|
| deduction / derivation | "Do the listed premises suffice to establish [@conclusion]?" |
| deduction / observation | "Is the observation of [@conclusion] reliable under the stated conditions?" |
| deduction / computation | "Is the computation of [@conclusion] correctly implemented?" |
| infer | "Is the statistical association between [@hypothesis] and [@evidence] valid at the stated probabilities?" |
| equal (Operator) | "Are [@a] and [@b] truly equivalent?" |
| contradict (Operator) | "Do [@a] and [@b] truly contradict?" |

Templates are rendered to concrete questions using the referenced Claims' labels or content.

## 4. Compilation target summary

| Lang construct | IR compilation target |
|---|---|
| `Context(...)` | `Knowledge(type="context")` |
| `Setting(...)` | `Knowledge(type="setting")` |
| `Claim(...)` / subclasses | `Knowledge(type="claim")` + bound parameters |
| `Question(...)` | `Knowledge(type="question")` |
| `derive(...)` | `FormalStrategy(type="deduction")` + conjunction + implication helpers |
| `observe(...)` with given | `FormalStrategy(type="deduction", metadata.pattern="observation")` |
| `observe(...)` no given | `FormalStrategy(type="deduction", premises=[], metadata.pattern="observation")` + `Grounding(kind="source_fact")` on the Claim |
| `compute(...)` / `@compute` | `FormalStrategy(type="deduction", metadata.pattern="computation", metadata.compute={...})` |
| `equal(A, B)` | `Operator(type="equivalence")` + Equivalence helper Claim |
| `contradict(A, B)` | `Operator(type="contradiction")` + Contradiction helper Claim |
| `infer(...)` | `Strategy(type="infer", premises=[H], conclusion=E)` + CPT `[p_e_given_not_h, p_e_given_h]` + StatisticalSupport helper |
| `given=(A, B, C)` tuple | `Operator(type="conjunction")` + conjunction helper Claim |
| `rationale=` | `steps=[Step(reasoning=rationale)]` |
| `background=` | `Strategy.background` |
| Review | `ReviewManifest` with `Review` entries |

## 5. Schema change summary

### 5.1 Modified

- `Knowledge.type` adds `"context"` value
- `Parameter.value: JsonValue | None` — bound parameter values (including QID references)

### 5.2 New metadata conventions

- `metadata.grounding` — root Claim provenance (kind, rationale, source_refs)
- `metadata.content_template` — original docstring template before rendering
- `metadata.action_label` — Lang Action label, QID-style (`{ns}:{pkg}::action::{label}`)
- `metadata.pattern` — observation/derivation/computation distinction
- `metadata.compute` — computation function_ref, code_hash, etc.

### 5.3 New types

- `ReviewManifest` — package-level review state
- `Review` — individual Strategy/Operator review record

### 5.4 Unchanged

- `Strategy` schema — no new fields
- `StrategyType` enum — no new types
- `Operator` schema — no changes
- `FormalStrategy` / `FormalExpr` — no changes
- `Step` — no changes
- `StrategyParamRecord` — no changes (infer CPT already sufficient)
- Package, Module structure — no changes

## 6. Open questions

1. Should `CompositeStrategy` be kept in IR for future composition support?
   Recommended: keep for forward compatibility, but v6 does not use it.

2. Should `exhaust()` (exhaustive alternatives) be added?
   Recommended: defer, requires disjunction + mutual exclusion combined semantics.

3. Should legacy `StrategyType.SUPPORT` be deprecated in favor of `DEDUCTION`?
   Recommended: keep both for now, deprecate `SUPPORT` in v7.
