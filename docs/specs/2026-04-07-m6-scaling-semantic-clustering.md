# M6 Scaling Semantic Clustering — Infra Design

> **Status:** Draft
> **Date:** 2026-04-07
> **Scope:** 从 100 万扩展到 1 亿+ embedding 的分层聚类架构
> **前置:** `2026-04-04-m6-semantic-discovery.md`（当前单机实现）

## 概述

当前 M6 Semantic Discovery 使用 FAISS IndexFlatIP（精确搜索）+ Union-Find，单机运行。适用于 <50 万规模。本文档设计扩展到 1 亿+ 的分层聚类架构。

**核心方案：两阶段 top-p 分区 + 分布式精确搜索**

- 阶段 0：k-means 训练聚心，每个向量分配到 top-p 个最近聚心（分区）
- 阶段 1：按分区导出 embedding 到 S3
- 阶段 2：每个分区内独立精确搜索 + Union-Find（多机并行，任务队列调度）
- 阶段 3：全局合并去重

**关键设计决定：**
- 与 scaling BP 的图分区方案独立——BP 按图拓扑分区（最小化跨分区边），聚类按向量空间分区（最大化相似向量同分区概率），两者目标不同
- ByteHouse 作为 source of truth（写入端），S3 作为计算快照（读取端）
- 离线批处理，允许小时级延迟

## 算法：两阶段 top-p

### 为什么不用简单 k-means 分区

简单 k-means 把每个向量分配到 1 个最近聚心。两个相似向量如果在聚心边界上，会被分到不同分区而漏检。

### top-p 方案

每个向量分配到距离最近的 **p 个**聚心（默认 p=3）。两个相似向量只要共享至少 1 个分区就能被发现。p=3 时 recall >95%。

**流程：**

```
1. 训练 K 个聚心（k-means，可采样训练）
2. 每个向量和 K 个聚心比较，取 top-p 个 → 分配到 p 个分区
3. 每个分区收集所有被分配到它的向量 → 独立精确搜索
4. 全局去重（同一个 pair 可能在多个分区都被发现）
```

### 对比其他策略

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 相邻分区交叉搜索 | 每个分区和 m 个邻居分区合并搜索 | 简单 | 分区不独立，需要邻居信息；增量时每个新向量搜 m+1 个分区 |
| 边界带冗余 | 距离差 < δ 的向量冗余到多个分区 | 分区独立 | δ 全局固定，无法适应不同密度区域 |
| **top-p（推荐）** | 每个向量取 top-p 个最近聚心 | **分区独立，p per-vector，增量 <1ms/向量，聚心过时容错好** | 需要去重，存储冗余 ~p 倍 |

## 参数选择

### 核心参数

| 参数 | 推荐值 | 含义 |
|------|--------|------|
| K | N / 5,000（向上取整） | k-means 聚心数 |
| p | 3 | 每个向量分配到的分区数 |
| similarity_threshold | 0.85 | Union-Find 合并阈值 |
| faiss_k | 100 | 分区内 k-NN 搜索邻居数 |

### 有效分区大小

每个分区的实际向量数 = N × p / K ≈ 15,000（当 K = N/5000, p=3）。

### K 的 tradeoff

**增大 K**（更多更小的分区）：
- Pro：分区内搜索更快（n² 下降）
- Pro：更多分区 → 更大并行度
- Con：k-means 训练变慢（O(N × K × d × iters)）
- Con：分区越小，相似向量被切到不同分区的概率越高，更依赖 p 的冗余
- Con：更多 S3 文件管理开销

**减小 K**（更少更大的分区）：
- Pro：相似向量大概率在同一分区，recall 高
- Pro：k-means 训练快
- Con：分区内搜索变慢
- Con：并行度受限

**推荐范围**：有效分区大小 5,000-20,000。具体取决于可接受的搜索时间（~1s per 15k 分区）和机器数量。

### p 的 tradeoff

| p | 冗余倍数 | 预期 recall | 搜索总量倍数 |
|---|---------|------------|------------|
| 1 | 1x | ~85% | 1x |
| 2 | 2x | ~93% | 2x |
| 3 | 3x | >95% | 3x |
| 5 | 5x | >99% | 5x |

p=3 是性价比最高的选择。

## 存储方案

ByteHouse 写入 + S3 读取。

| 数据 | 存储 | 格式 | 理由 |
|------|------|------|------|
| Embedding 向量 | ByteHouse `node_embeddings` | Array(Float32) | Source of truth，增量 INSERT |
| 聚心 | S3 | `.npy` / `faiss.write_index` | Worker 加载，不常变 |
| 分区分配 | ByteHouse `partition_assigns` | gcn_id → partition_ids (Array) | SQL 可查，增量更新 |
| 分区快照 | S3 | 每分区一个 `.npy` | Worker 按需下载，批量计算读取快 |
| 聚类结果 | ByteHouse `discovery_clusters_v2` | 已有表结构 | SQL 查询，和现有系统一致 |
| 任务状态 | Redis / ByteHouse | 分区 ID → 状态 | 队列调度 |

**数据流：**
```
增量写入: embedding → ByteHouse (INSERT)
批量聚类: ByteHouse → 导出分区到 S3 → worker 从 S3 读 → 结果写 ByteHouse
```

ByteHouse 擅长写入和 SQL 查询，S3 擅长大文件顺序读取。各取所长。

## 增量更新

### 新向量处理

```
新向量 v_new 进来:
1. 和 K 个聚心比较，找 top-p → O(K × d) < 1ms
2. 写入 ByteHouse partition_assigns
3. 累积到触发条件后，只对有新向量的分区重新搜索
```

### 触发策略

- 累积 10,000 个新向量，或
- 距上次增量聚类 24 小时
- 以先到者为准

### 聚心过时检测

监控：新向量到最近聚心的平均距离。

```
训练时的平均距离: d_train
当前新向量平均距离: d_current

if d_current > d_train * 1.2:
    触发全量 k-means 重训练
```

预计频率：每月或每增加 20% 新数据时。

重训练不停机：用新数据采样训练新聚心 → 重新分配 → 切换。旧索引保留到新索引就绪。

## 任务调度

### 架构

```
┌──────────┐     ┌─────────┐     ┌──────────┐
│ 协调器   │────▶│ 任务队列 │────▶│ Worker 1 │
│          │     │ (Redis)  │────▶│ Worker 2 │
│ 负责:    │     │          │────▶│ Worker 3 │
│ - 建队列 │     │ 每个任务:│     │ ...      │
│ - 导出S3 │     │ partition│     │ Worker N │
│ - 收结果 │     │ _id      │     └──────────┘
└──────────┘     └─────────┘
                                  每个 Worker:
                                  1. 从队列领取 partition_id
                                  2. 从 S3 下载 partition_{id}.npy
                                  3. FAISS 精确搜索 + Union-Find
                                  4. 结果写 ByteHouse
                                  5. 标记任务完成
```

### 任务幂等性

每个分区任务可以安全重试：
- 分区数据是只读快照（S3 文件不变）
- 结果写入用 `run_id + partition_id` 做 unique key，重复写入幂等
- Worker 无状态，任何 worker 可以处理任何分区

### 全局合并

所有分区完成后，协调器执行全局合并：
- 读取所有分区的聚类结果
- 去重：同一个 (gcn_id_a, gcn_id_b) pair 在多个分区被发现 → 只保留一个
- Union-Find 跨分区合并：如果分区 A 发现 (x, y) 同簇，分区 B 发现 (y, z) 同簇 → x, y, z 合并
- 写入最终 ClusteringResult

## 容错与可观测

### Worker 容错

- 任务超时：单分区预期 <10s，超过 5 分钟标记失败重新入队
- 最大重试 3 次，之后标记为永久失败，人工介入
- Worker 进程异常退出 → 任务自动回到队列（visibility timeout 机制）

### 可观测指标

| 指标 | 含义 | 告警阈值 |
|------|------|---------|
| 任务队列 pending/completed/failed | 进度 | failed > 5% |
| 单分区搜索耗时 | 性能 | p99 > 30s |
| 全量聚类总耗时 | SLA | > 预期 2 倍 |
| 新向量平均聚心距离 | 聚心质量 | > 训练时 1.2 倍 |
| 分区大小分布标准差 | 分区均匀性 | 最大分区 > 平均 5 倍 |
| 聚类数量和大小分布 | 结果质量 | 异常突变 |

### 日志

- 每个阶段的开始/结束时间和关键计数
- 每个 worker 的 per-partition 处理日志
- 全局合并的去重统计
